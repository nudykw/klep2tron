# Actor (Игрок/NPC)

## 1. Требования
1.1. **Редактор моделей**: Наличие инструмента для визуальной настройки, разделения и кастомизации.
1.2. **Модульность**: Фигура должна состоять из трех заменяемых частей: голова, тело, двигатель.
1.3. **Кастомизация**: Возможность динамической смены мешей частей, их цветов и текстур во время игры.
1.4. **Управление**: Поддержка ввода с клавиатуры, геймпада и сенсорного экрана.
1.5. **Физика**: Перемещение по поверхности комнаты с учетом высоты тайлов пола и коллизий.
1.6. **Независимость частей**: Голова должна вращаться независимо от направления движения тела.
1.7. **Классы для хранения спрайта**: Каждая одна из трех частей должна хранится в отдельном классе. Он должен содержать информацию об весе, броне для всех частей тела, уникальные свойства типа энергии (уровень заряда батареи для тела, мощность двигателя для нижней части). Голова будет содержать оружие (характеристики, сенсоры).

## 2. Исследования
2.1. **Как будем разбивать модель на части?**
    *   2.1.1. Первичное разделение и редактирование будет происходить во внутреннем редакторе (Character Editor).
    *   2.1.2. Программное разделение по Y-координатам (анализ вершин) с возможностью визуальной корректировки границ.
2.2. **Как должны храниться файлы модели?** (Вердикт)
    *   2.2.1. Использование иерархической структуры папок в `assets/NPC/`:
        *   `Sources/`: Исходные файлы моделей (оригинальные `.obj`, `.glb`).
        *   `Meshes/`: Разделенные части тела (`Head/`, `Body/`, `Engine/`).
        *   `Textures/`: Текстуры, разделенные по частям тела.
        *   `Configs/`: `.json` файлы с "рецептами" сборки (ссылки на меши, текстуры, параметры веса и брони).
    *   2.2.2. Использование отдельных файлов `.obj` для каждой части для максимальной модульности.
    *   2.2.3. Хранение пресетов конфигурации в формате `.json`.
2.4. **Ввод в Bevy**: Использование ресурсов `ButtonInput<KeyCode>`, `Axis<GamepadAxis>` и `Touches`.
2.5. **Кинематика**: Реализация перемещения через `Transform` с ручной проверкой границ и высоты пола (на основе данных из `Project/Room`).

## 3. Общий план реализации
3.1. **Этап 1: Создание Редактора (Model Editor)**
    *   3.1.1. Вьюер моделей с Orbit Camera.
    *   3.1.2. Инструментарий для разделения единого меша на части по Y-координатам.
    *   3.1.3. Экспорт/сохранение разделенных частей и их настроек.
3.2. **Этап 2: Подготовка и загрузка**
    *   3.2.1. Расширение `ClientAssets` для работы с модульными частями.
3.3. **Этап 3: Логика Актора**
    *   3.3.1. Реализация иерархии сущностей и компонентов характеристик (вес, броня и т.д.).
3.4. **Этап 4: Системы управления и физики**
    *   3.4.1. Многоплатформенный ввод и перемещение с учетом окружения.
3.5. **Этап 5: Кастомизация и независимый поворот**
    *   3.5.1. Финализация систем вращения головы и смены материалов в рантайме.
